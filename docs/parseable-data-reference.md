# Parseable Data Reference

What information we can extract from Claude Code JSONL session logs, its reliability, and what visualization state machines it can drive.

## Data Source

Claude Code writes session logs to `~/.claude/*.jsonl` and `~/.claude/projects/**/*.jsonl`. Each line is a JSON object with a `type` discriminator field. We watch these files and process new lines as they appear.

---

## Event Types & Parseable Fields

### 1. Session Lifecycle

#### `session_start`
| Field | Type | Reliability | Notes |
|-------|------|-------------|-------|
| `sessionId` | string | High | Unique session identifier |
| `model` | string | High | e.g. `claude-opus-4-6` |
| `timestamp` | number | High | Epoch ms |

**Can drive:** Farm initialization, land allocation, model-specific theming.

#### `session_end` / `result`
| Field | Type | Reliability | Notes |
|-------|------|-------------|-------|
| `total_cost_usd` | number | High | Final session cost |
| `timestamp` | number | High | Session duration derivable from start/end |

**Can drive:** Harvest/wrap-up animation, session summary, final state snapshot.

---

### 2. Assistant Messages (type: `assistant`)

These are the richest events for context window tracking.

| Field | Type | Reliability | Notes |
|-------|------|-------------|-------|
| `message.usage.input_tokens` | number | **Authoritative** | Actual tokens sent to model |
| `message.usage.output_tokens` | number | **Authoritative** | Tokens generated by model |
| `message.usage.cache_read_input_tokens` | number | **Authoritative** | Tokens served from prompt cache |
| `message.usage.cache_creation_input_tokens` | number | **Authoritative** | Tokens written to prompt cache |
| `message.content` | array | Medium | Response content blocks (text, tool_use) |
| `model` | string | High | Model used for this response |
| `cost_usd` | number | High | Cost for this specific turn |

**Derived values:**
- `cumulativeInputTokens` — running sum across session
- `cumulativeOutputTokens` — running sum across session
- `cumulativeCost` — running USD total
- `fillPercentage` — `inputTokens / contextWindowMax * 100`

**Can drive:** Context capacity gauge (the primary "how full is the farm" indicator), cost tracking, per-turn consumption rate, cache efficiency visualization.

**Important:** `input_tokens` is the **total** context sent to the model on that turn — it includes conversation history, system prompt, tool results, everything. This is the single most reliable number for "how full is the context window." However, it's a lump sum — we cannot decompose it into individual items.

### 3. Tool Use (type: `tool_use`)

| Field | Type | Reliability | Notes |
|-------|------|-------------|-------|
| `toolName` | string | High | Tool identifier |
| `toolInput` | object | High | Tool-specific arguments |
| `messageId` | string | High | Correlates with tool_result |
| `timestamp` | number | High | When tool was invoked |

#### Known Tool Inputs

**Read:**
- `toolInput.file_path` (string) — file being read
- `toolInput.offset` (number, optional) — start line
- `toolInput.limit` (number, optional) — max lines

**Write:**
- `toolInput.file_path` (string) — file being written
- `toolInput.content` (string) — new file content

**Edit:**
- `toolInput.file_path` (string) — file being edited
- `toolInput.old_string` (string) — text being replaced
- `toolInput.new_string` (string) — replacement text

**Bash:**
- `toolInput.command` (string) — shell command
- `toolInput.timeout` (number, optional) — timeout ms

**Glob:**
- `toolInput.pattern` (string) — glob pattern
- `toolInput.path` (string, optional) — search root

**Grep:**
- `toolInput.pattern` (string) — search regex
- `toolInput.path` (string, optional) — search root
- `toolInput.glob` (string, optional) — file filter

**Agent (subagent spawn):**
- `toolInput.prompt` (string) — task description
- `toolInput.subagent_type` (string) — agent type

**WebSearch:**
- `toolInput.query` (string) — search query

**WebFetch:**
- `toolInput.url` (string) — URL to fetch
- `toolInput.prompt` (string) — extraction prompt

**AskUserQuestion:**
- `toolInput.questions` (array) — questions for user

**MCP / Custom tools:**
- `toolName` will be an arbitrary string (e.g. `mcp__server__toolName`)
- `toolInput` has unknown schema — treat generically

**Can drive:** Character action state (investigating, writing, running commands), file-specific plot activity, tool-type animations, subagent spawning.

### 4. Tool Result (type: `tool_result`)

| Field | Type | Reliability | Notes |
|-------|------|-------------|-------|
| `toolName` | string | High | Matches corresponding tool_use |
| `toolResult` | unknown | High | Raw output (string, object, etc.) |
| `messageId` | string | High | Correlates with tool_use |
| `error` | string | Medium | Present on failure |

**Derived values:**
- `isError` — whether the tool call failed
- `resultLength` — string length of result → rough token estimate (`length / 4`)
- `file_path` — carried over from correlated tool_use via messageId

**Can drive:** Plot token size estimates, error/bug spawning, success/failure animations, crop growth (for file reads).

## Authoritative vs. Estimated Data

### Authoritative (from the API response)
| Data Point | Source | Use |
|------------|--------|-----|
| Total input tokens per turn | `message.usage.input_tokens` | Context capacity — this IS the fill level |
| Total output tokens per turn | `message.usage.output_tokens` | Generation cost |
| Cache read tokens | `message.usage.cache_read_input_tokens` | Cache efficiency |
| Cache write tokens | `message.usage.cache_creation_input_tokens` | Cache cost |
| Cost per turn | `cost_usd` | Running cost |
| Model name | `model` | Model-specific behavior |

### Estimated (heuristic-derived)
| Data Point | Method | Accuracy |
|------------|--------|----------|
| Per-file token count | `content.length / 4` | Rough (~75% accurate) |
| File presence in context | "Was it read?" tracking | Degrades over time — no eviction signal |
| Queue position / recency | Turn-based ordering | Accurate for relative order, not for actual eviction |
| Context composition | Sum of file estimates | Undercounts — misses conversation, system prompt, tool overhead |

## Context Composition Breakdown (via `/context` command)

Claude Code's `/context` command provides a **per-category token breakdown** of the current context window. This is not available in JSONL logs, but represents what a fully-observable system could provide. It serves as our reference for what categories of context exist and their typical proportions.

### Categories

| Category | Description | Typical % | Observable in JSONL? |
|----------|-------------|-----------|---------------------|
| **System prompt** | Claude Code's built-in instructions | ~2-4% | No — injected silently |
| **System tools** | Built-in tool definitions (Read, Write, Edit, Bash, etc.) | ~8-10% | Partially — we know the tool list from `tool_use` events, but not the token cost of their definitions |
| **MCP tools** | Tool definitions from connected MCP servers | ~5%+ | Partially — we see MCP tool names when called, but not their schema definitions' token cost |
| **Custom agents** | User-defined agent type definitions | <1% | No — loaded from config |
| **Memory files** | CLAUDE.md, auto-memory, project memory | <1% | No — injected silently |
| **Skills** | Registered slash command definitions | <1% | No — loaded from config |
| **Messages** | Conversation history (user + assistant + tool results) | Grows over session | Partially — we see tool results and can estimate, but not user messages or assistant text token counts individually |
| **Free space** | Unused context capacity | Shrinks over session | Derived — `maxTokens - input_tokens` |
| **Autocompact buffer** | Reserved space that triggers compression when consumed | ~16-17% | No — internal threshold |

### Key Observations

1. **Fixed overhead is significant.** System prompt + system tools + MCP tools + memory + skills can consume 15-20% of context before any conversation begins. This is invisible in JSONL logs.

2. **MCP tools scale with connections.** Each MCP tool definition costs ~292 tokens. A user with 34 MCP tools (as seen in the example) consumes ~10k tokens just on tool definitions. More MCP servers = less room for actual work.

3. **Autocompact buffer is pre-allocated.** ~16.5% of the context window is reserved as a buffer — when messages encroach on this space, auto-compaction triggers. The effective usable context is `maxTokens * ~0.835`.

4. **Messages are the only growing category.** Everything else is roughly fixed per session. The visualization's dynamic element should focus on message growth vs. remaining space.

### What This Means for Visualization

The farm's total land area represents `maxTokens`. But a significant portion is "infrastructure" that's always there:
- System prompt, tools, MCP, memory, skills = **permanent structures** (buildings, wells, fences that are always present)
- Autocompact buffer = **reserved land** (a visual boundary the crops shouldn't cross)
- Messages = **the actual crops** (growing, being harvested, being compressed)
- Free space = **open soil**

This decomposition could drive the initial farm layout: infrastructure occupies fixed positions, and the dynamic crop area is what remains.

## Black Box Gaps (Claude Code Internals)

These are things that happen inside Claude Code that produce **no JSONL events**:

| Hidden Process | Impact | Workaround |
|----------------|--------|------------|
| System prompt injection | ~2-4% of context, unknown exact content | Assume fixed overhead; `/context` gives exact token count |
| System tool definitions | ~8-10% of context | Count from known tool list |
| MCP tool definitions | ~292 tokens per tool | Count connected MCP tools × 292 |
| Custom agent definitions | <1% of context | Assume small fixed overhead |
| Memory file injection | <1% of context | Known if user has CLAUDE.md |
| Skill definitions | <1% of context | Assume small fixed overhead |
| Autocompact buffer | ~16.5% reserved | Treat as hard boundary |
| Context compression/summarization | Old messages silently replaced with summaries | Monitor `input_tokens` drops between turns |
| Message eviction | Oldest messages dropped when context full | Infer from sudden `input_tokens` decrease |
| Auto-compact triggers | Fires when messages encroach on buffer | Watch for `input_tokens` < previous `input_tokens` |
| Tool result truncation | Large results silently trimmed to fit | Compare result length vs expected |

### Detecting Compression (Heuristic)

When `input_tokens` on turn N is **significantly less** than `input_tokens` on turn N-1 (despite adding new content), compression likely occurred. This is the closest we can get to detecting eviction without explicit events.

```
Turn 5: input_tokens = 45,000
Turn 6: input_tokens = 48,200  (normal growth)
Turn 7: input_tokens = 32,100  (compression detected — ~16k tokens evicted/summarized)
```

**Can drive:** Weather change (sunny → overcast), wilting/harvesting animations on oldest plots, "fog" over uncertain regions.

## Custom / MCP Tool Handling

Custom tools (MCP servers, user-defined skills) appear as:
- `toolName`: arbitrary string, often `mcp__serverName__toolName`
- `toolInput`: unknown schema
- `toolResult`: unknown schema

**What we can always extract:**
- That a tool was called (name, timestamp)
- Whether it succeeded or failed (error field)
- Raw result size (string length → token estimate)

**What we cannot extract without tool-specific parsing:**
- File paths, URLs, or other semantic content from inputs/results
- Whether the tool reads files, writes files, or does something else entirely

**Strategy:** Treat unknown tools as generic "activity" — the agent is doing *something*. Visualize as a generic action animation. If the tool name contains recognizable patterns (e.g., `file`, `read`, `write`, `search`), optionally attempt heuristic categorization.

## State Machine Input Summary

Each JSONL event type maps to state machine inputs:

| Event | SM Input | Drives |
|-------|----------|--------|
| `session_start` | `SESSION_BEGIN` | Farm initialization |
| `session_end` | `SESSION_END` | Farm wrap-up |
| `assistant` (with usage) | `TOKENS_UPDATED` | Capacity gauge, fill level, cost |
| `tool_use` (Read) | `FILE_READ_START` | Plot activity, character movement |
| `tool_result` (Read) | `FILE_READ_COMPLETE` | Plot sizing, crop growth |
| `tool_use` (Write/Edit) | `FILE_WRITE_START` | Plot activity, character movement |
| `tool_result` (Write/Edit) | `FILE_WRITE_COMPLETE` | Plot update |
| `tool_use` (Bash) | `COMMAND_START` | Character action animation |
| `tool_result` (Bash, error) | `ERROR_OCCURRED` | Bug spawn |
| `tool_use` (Glob/Grep) | `SEARCH_START` | Investigation animation |
| `tool_result` (Glob/Grep) | `SEARCH_COMPLETE` | Results visualization |
| `thinking` | `THINKING_START` | Processing animation |
| `subagent_start` | `SUBAGENT_SPAWN` | Secondary character spawn |
| `subagent_stop` | `SUBAGENT_DESPAWN` | Secondary character removal |
| `permission_request` | `WAITING_FOR_USER` | Idle/blocked state |
| `input_tokens` drop | `COMPRESSION_DETECTED` | Eviction/weather change |
| Any MCP/custom tool | `GENERIC_TOOL_START/COMPLETE` | Generic activity animation |
